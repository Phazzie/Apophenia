## 🛠️ Autonomous Coding System Design

To improve accuracy and reliability autonomously, we will implement the following system:

### 1. Seam-Validation System (`scripts/validate-seams.ts`)
A static analysis tool that runs before every commit.
*   **Purpose:** Ensures that all engine classes strictly adhere to the interfaces defined in `src/core/types/seams.ts`.
*   **Mechanism:** It scans the codebase, parses the AST (Abstract Syntax Tree), and verifies that:
    *   Every class implementing `BaseEngine` has a valid `process()` method.
    *   No engine mutates `store` state directly (must return `EngineOutput`).
    *   All `EngineOutput` objects match the schema.
*   **Benefit:** Prevents "drift" where an engine implementation diverges from the architecture.

### 2. Narrative Simulation Engine (`scripts/narrative-simulator.ts`)
A runtime verification tool that plays the game in "headless" mode.
*   **Purpose:** Detect runtime crashes and narrative dead-ends.
*   **Mechanism:**
    *   Initializes the game loop.
    *   Uses a "Random Agent" or "Heuristic Agent" to make choices.
    *   Runs for 50-100 turns per session.
    *   Logs any exceptions, undefined states, or broken UI renders.
*   **Benefit:** Catches "rare" bugs that only appear after long play sessions (e.g., memory leaks, state corruption).

### 3. Circuit Breaker Monitor (`src/services/monitoring/CircuitBreaker.ts`)
*   **Purpose:** Real-time health monitoring of AI providers.
*   **Mechanism:**
    *   Tracks error rates for Grok, Gemini, etc.
    *   If Error Rate > 20% over 1 min, Open Circuit (stop calling).
    *   After 5 mins, Half-Open (try one request).
*   **Benefit:** Prevents the game from freezing while waiting for a dead API.
